    Beaucoup de gens sont venus me voir pour des explications sur les images de la MinilibX.
    


    Je voudrais donc écrire un mini-manuel ici pour que vous puissiez commencer plus facilement.
    


    


     Les images MLX
    


    1) A quoi ça sert ?
    


    Si vous avez bien vu la vidéo (sinon allez y de suite !) vous avez bien vu qu'on pouvait imprimer un pixel a l'écran à l'aide de la fonction mlx_put_pixel. Super. Du coup, a quoi bon se faire chier à utiliser autre chose ?
    


    Et bien simplement pour des histoires de performance. Quand vous faites tourner un programme qui imprime dans la console des centaines de printf, vous avez pu vous rendre compte que ça ralentissait énormément le programme. Imprimer tous les pixels un par un c'est un peu pareil.
    


    Solution : ''imprimer'' tous les pixels dans la mémoire puis tout imprimer sur l’écran en une fois, et c'est là que les images servent à quelque chose. L'image MLX c'est simplement le truc qui contient tout vos pixels en mémoire.
    


    L'utilisation des images n'est pas obligatoire pour un FdF simple, mais à partir de Fractol on vous demande un affichage qui se modifie en temps réel et vous devrez donc forcément utiliser les images.
    


    2) Comment ça fonctionne ?
    


    Les images de la MLX sont en fait simplement... des strings (des char * terminés par \0, quoi). 1 image = 1 string.
    
EDIT: Cf commentaire de Ol ci-dessous, ce sont en fait des suites de char non terminées par \0 donc techniquement pas des strings
    


    Chaque pixel de votre image est placé a la suite dans le string, en commençant par le pixel (0, 0) en haut a gauche, puis vers la droite, puis vers le bas. Chaque pixel est décrit par 4 chars. 
    


    Exemple pour une résolution de 800x600 : Vous avez 800x600 = 480 000 pixels.
    
Votre string fera donc 1 920 000 caractères (+ le \0 final).
    


    Comment trouver le bon pixel, dans ce cas ? C'est trés simple : partons du premier pixel, donc tout en haut a gauche, qui est le pixel (0, 0) sur votre écran, donc le pixel 0 dans votre string.
    


    Imaginons que vous voulez changer le 6ème pixel, c'est à dire le (5, 0), c'est a dire qu'on se décale de 5 vers la droite. NB: par convention on écrit toujours (x, y) avec x = abscisses et y = ordonnées
    


    Votre pixel est donc décrit par la position dans le string : 0 + 4 * 5 = 20, sur 4 chars, donc il s'agit des positions 20, 21, 22, 23.
    


    Comment descendre sur l'axe des Y maintenant ? Très simple : comme on revient a la ligne a chaque fois, pour passer de (0, 0) a (0, 2) on fait simplement 0 + 4 * Taille de la ligne * 2. Ici, sur une resolution de 800x600, la taille de la ligne est de 800 pixels. Donc le pixel (0, 2) est decrit par les chars 6400, 6401, 6402, 6403 sur notre string.
    


    Autre question : pourquoi 4 chars par pixel ? Simplement : le char 1 est la valeur de BLEU, le deuxieme de VERT, le troisieme de ROUGE => RGB. Le quatrième représente l'alpha, ignorez le pour le moment. Chaque char contient en fait une valeur d'int de 0 a 255. Le pixel décrit par (0, 0, 255, alpha) sera donc tout rouge. Le pixel (255, 255, 255, alpha) sera tout blanc.
    


    3) Comment ça s'utilise en pratique ?
    


    C'est un poil compliqué à mettre en place, mais la bonne nouvelle c'est que vous pourrez juste copier/coller votre code pour tout les projets graphiques.
    


    Etape 1 : créer une image
    


    Créer une image avec la fonction associée :
    


    pointeur_image = mlx_new_image(pointeur_mlx, largeur, hauteur);

    


    Gardez bien dans un coin le pointeur image (void*) vous en aurez besoin.
    


    Etape 2 : Obtenir le string de l'image
    


    my_image_string =  mlx_get_data_addr(pointeur_image, &(bpp), &(s_l), &(endian));

    


    Pour information, les variables envoyees en adresse ici (parametres 2 3 et 4) signifient :
    
nombre de Bit par pixel, size_line (taille de la ligne), et endian.
    
Ces variables doivent être envoyées a la fonction qui les initialise à la bonne valeur.  Personnellement, je ne les utilise pas par la suite. Vous pouvez utiliser size_line pour votre calcul de position de pixel, mais je suis feignant et peureux donc perso je fais autrement.
    


    Addendum facultatif a propos de ces parametres :  je reprécise un peu car pas mal d'incompréhensions ici : vous devez évidemment déclarer bpp, s_l, endian, mais pas besoin de les initialiser, leurs valeurs seront données par la fonction mlx_get_data_addr, c'est pourquoi on a besoin de lui envoyer l'adresse des variables et non pas leurs valeurs (qui n'existent pas si vous ne les avez pas initialisées). L’intérêt de passer ces variables a la fonction est de récuperer les bonnes valeurs pour s'en servir plus tard dans votre programme, mais dans la pratique on peut se débrouiller sans.
    
Pour info les valeurs sont initialisées comme suit par la fonction, de mémoire donc je peux me tromper : bpp = 4 chars de 8 octets = 32, size_line = longueur de votre resolution (pour du 800x600 c'est 800 fois 4) et endian est egal a 0 ou 1 selon la machine, sur les macs de l'ecole je crois que c'est 0.
    


    Etape 3 : Remplir l'image
    


    Je ne vous dit pas comment tout faire ici, je vous laisse un peu chercher ^_^ mais ce que je vous conselle, c'est de créer une fonction dans le genre : 
    


    void fill_pixel(char *my_image_string, int x, int y, int color);

    


    Qui va ''poser'' la couleur sur le pixel aux positions x et y. Je vous ais déjà détaillé les calculs plus haut pour ça.
    


    Etape 4 : Afficher l'image
    


    Juste:
    


    mlx_put_image_to_window(data, pointeur_window, pointeur_image, 0, 0);

    


    les deux derniers int sont la position ou vous voulez mettre votre image dans votre fenêtre, en général vous voudrez laisser à 0, 0.
    


    ''data'' c'est votre structure environnement (si vous ne savez pas ce que c'est : en gros c'est une structure ou vous foutez vos variables utiles, genre les pointeurs de la minilibX par exemple, ce qui est essentiel dans les projets graphiques pour avoir tout sous la main). Je ne suis plus certain si la fonction utilise le pointeur MLX ou le pointeur Window que vous aurez dans votre structure data, donc je vous laisse vérifier ce détail si vous le voulez. Il s'agit probablement du pointeur MLX vu qu'on envoie le pointeur window après.
    


    Comment obtenir des animations, par exemple pour l'ensemble de Julia dans Fract'ol ?
    


    Dans votre boucle d'events, faites en sorte de redessiner entièrement votre image après avoir changé vos paramètres, puis réaffichez l'image avec mlx_put_image_to_window. Si vous voulez être propre, détruisez l'image avant avec mlx_destroy_image.

    J'en profite pour dire que une image de la mlx n'est pas une string.

    
Une string c'est une suite continue de char dont on a l'adresse du premier et qui par convention ne contient pas de '\0' sauf pour en indiquer la fin.

    
 L'image de la mlx c'est simplement une suite continue de char dont on connait l'adresse du premier. Il se trouve que vous pouvez calculer la taille avec size_line et le nombre de ligne utilisé lors de la creation de l'image. Mais il n'y a rien au dela de cette taille (a part peut-etre le segfault). Il n'y a pas de char special pour indiquer la fin. Et vous pouvez avoir plein de '\0' dans les pixels de votre image.

    
Un "char *" c'est un pointeur sur char, donc ca contient l'adresse en mémoire d'un caractère. C'est tout. C'est votre contexte qui en fait eventuellement une string. Mais rien de systématique.

    caster le "image_string" en unsigned int * est aussi une bonne idee, il permet de se deplacement bien plus facilement dans l'image, de faire moin d'ecritures en memoire aussi (4 par 4 au lieux de un par un pour les couleurs) , le reste se fait aux operateurs binaires par exemple: https://github.com/Chr0nos/draw/tree/master/blend131 pour les blends modes, chaque composante etant comprise entre 0x0 et 0xff
    


    pour acceder a un pixel:
    


    

    pixels[y * width + x] = color;
    

    


    ou width = la largeur de l'image
    


    x = position horizontale
    


    y = position verticale
    


    evidement cela ne vaut que dans le cas d'images RGBA avec 32 bits par pixel ( 8 pour chaque couleur + 8 pour la couche alpha )
    


    si vous faites une lib pour vos fonctions graphiques je ne saurais que vous deconseiller de les prototyper sur des structures "a la mord moi le noeud" genre un "t_my_mlx" , mais bien de rester le plus generique possible en prevision du changement de lib
